# Plan: Error Boundary Wrapper for Entire Project

## Goal
Add a global error boundary in `App.tsx` with per-route error boundaries so all pages gracefully catch render errors instead of crashing to a blank white screen. Includes a per-route retry mechanism and a consistent fallback UI that matches the existing design language.

## Context
- React 18.3.1 (class-based `ErrorBoundary` required — `componentDidCatch` is the only way to catch render errors in React)
- No existing error boundary component anywhere in the project
- No `react-error-boundary` package installed (unnecessary — a 40-line class component handles it)
- Project uses `@/` path aliases, Tailwind CSS variables (`var(--background)`, `var(--accent)`, etc.), and `bento-card` pattern
- `App.tsx` has a `showLayout` boolean splitting routes into two groups: public (no layout) and dashboard (with sidebar/header)

## Design Decisions
1. **Write a class component by hand** (not `react-error-boundary`) — it's ~40 lines, 0 dependencies, and React's `componentDidCatch` is stable
2. **Two-layer strategy**:
   - **Global boundary** in `App.tsx` wrapping the entire `<AppContent />` — catches state corruption, unexpected nulls, etc.
   - **Per-route boundary** wrapping every `<Route element={...}>` — catches individual page render errors without crashing the whole app
3. **Fallback UI** matches existing design: uses `var(--card)`, `var(--accent)`, `lucide-react` `AlertTriangle` icon, and a "Try Again" button that calls `this.setState` to re-mount
4. **Per-route boundaries reset on route change** so navigating away and back auto-retries

## Implementation Steps

### Step 1: Create `src/app/components/ErrorBoundary.tsx`
- Class component with `state: { hasError: boolean; error: Error | null }`
- `static getDerivedStateFromError(error)` → sets `{ hasError: true, error }`
- `componentDidCatch(error, errorInfo)` → logs to console
- `handleRetry()` method calls `this.setState({ hasError: false, error: null })`
- Renders:
  - If `hasError`: a centered card with `AlertTriangle` icon, error message (or generic friendly text), "Try Again" button
  - Otherwise: `this.props.children`
- Accept `fallback?: React.ReactNode` prop for custom fallback per context
- Accept `children: React.ReactNode` prop
- Use `@/` import for consistency

### Step 2: Update `App.tsx` — wrap in global + per-route boundaries
- Import `ErrorBoundary` from `@/app/components/ErrorBoundary`
- **Global**: Wrap the `BrowserRouter` children: `<ErrorBoundary><AppContent /></ErrorBoundary>`
- **Per-route (showLayout=true routes)**: Wrap each `<Route element={...}>` in its own `<ErrorBoundary>` so one page crash doesn't take out the sidebar/header
- **Per-route (public routes)**: Wrap `/`, `/auth`, `/onboarding`, `/loading` each in `<ErrorBoundary>`
- The global boundary in the `App()` function wraps everything, but set `key={location.pathname}` on each per-route boundary so navigation resets the error state

### Step 3: Verify build
- Run `npm run build` to confirm 0 errors
- Verify the component catches errors gracefully

## Edge Cases Covered
- Render errors inside individual pages → caught by per-route boundary, only that page shows fallback
- State corruption in `useFinPathStore` or `useAuthStore` → caught by global boundary
- Navigation after error → `key={location.pathname}` resets per-route boundary state
- Retry → `handleRetry` resets error state, React re-renders the children
- Nested errors → global boundary catches anything not caught by per-route boundaries
- Sidebar/Header/PennyPanel errors → they run in the `showLayout` tree, caught by global boundary (intentional — layout components failing is a full-app problem)

## What This Does NOT Cover
- Async errors in event handlers (these don't trigger error boundaries — existing `try/catch` in event handlers handles these)
- Network errors from API calls (already handled with `try/catch` in fetch calls)
- React Router 404 errors (handled by the current redirect logic)
